Detailed Specification of 'QuickenPrices' App

Core Purpose
The app fetches and processes stock prices and exchange rates, converts currencies where necessary, logs detailed activity, and generates a Quicken-compatible CSV file. It also automates importing the CSV into Quicken and provides structured, human-readable terminal outputs summarizing its operations.

Major Features and Functionalities

1. Technical considerations
Written in python
Uses yfinance library to access the yahoo finance api
Exchange rates and prices must be date matched precisely. Data comes from different time zones so all dates must be converted to UTC and then normalised and formatted to iso standard before use. 
Use Decorators for Cross-Cutting Concerns.
Excellent separation of concerns and good use of classes.
A standard error handling, retry mechanism and logging mechanism for debug, info, warning and error levels must be defined, ensuring they are modular and reusable across the application and used consistently where appropriate.

Error Handling:

Centralized handling of expected and unexpected exceptions.
Detailed logging for traceability.
Graceful degradation: Where possible, fallback mechanisms instead of crashing.

Logging System:

Setup a CentralizedLogger with two functions a rotating log file handler and Console handler (called terminal handler) 
The logger provides color-coded output for improved readability.
Configurable logging levels (DEBUG, INFO, WARNING, ERROR) defined from YAML for log file handler and terminal handler.
Distinct formatting and content for terminal handler output and logfile content. Debug for log includes operation and code line numbers.
Use of a decorator @log_function: Logs function entry, exit, and execution details. Logs the start and end of functions at DEBUG level. Captures arguments, results, and execution time for debugging.
don't use @log_function on CentralizedLogger but use the internal python logger.

Retry Mechanism:

A decorator @retry: Implements retry mechanism with logging for transient errors.
Wraps functions to handle transient errors.
Configurable maximum retries and delay between retries (from YAML).
Logs each retry attempt and eventual success or failure.
Useful for operations prone to transient failures, such as API calls.

2. Configuration Management
Never hard code settings. All configurable parameters, such as logging levels, retry counts, or file paths, will be read from the config.yaml file. If settings are missing, meaningful defaults should be provided but logged as a fallback. Always look for settings or include them in a user-provided config.yaml file, which includes:

Tickers: A mix of stock tickers, indices, FX pairs, and commodities are listed. (e.g., AMGN, ^FTSE, GBP=X, BZ=F ).
Debugging: Specific tickers are flagged for additional debugging.
Paths: Clearly defined paths for base directory, Quicken executable for automation, CSV File (Outputs  price data for Quicken import), Log File to record detailed logs and cache directory to store previously fetched data.
Collection Settings: Includes parameters for fetching data (e.g., Period for fetching price/ exchange history (period_years), retry mechanism).
Cache Settings: Parameters to manage the cache's lifespan and cleanup.
API Rate Limits: Configured to prevent overuse of the Yahoo Finance API.
Logging configuration: Log levels for terminal and file outputs, content and formatting of log levels and file rotation settings (e.g., max file size, backup count).
Currency Pairs: Comprehensive list of currency exchange pairs to access exchange rates and descriptions of the conversion.

3a Stock Price Updates
Fetching Stock Prices:
Retrieves historical prices for each ticker over a  period defined in yaml (e.g., 1 month). 
Uses the closing price for all days other than today. Uses the latest available price for today.
Handles both valid tickers and invalid or delisted tickers gracefully.
Adjusts for non-trading days or missing data by skipping unavailable dates.
distinguishes between stock tickers, indices tickers "^...", FX rate tickers "...=X" and commodity tickers "...=F".

Currency Handling:
Extracts the currency of each stock ticker (e.g., USD, GBp) using metadata.
Converts stock ticker non-GBP prices to GBP using live or cached exchange rates. Indice tickers "^...", FX rate tickers "...=X" and commodity tickers "...=F" all remain unchanged.
Adjusts stock tickers priced in GBp by dividing prices by 100 to convert them to GBP.
Statuses:
Outputs the status of each ticker (e.g., ✓ New Data, # Cached, ✗ Failed) separated into stock, indice, FX rate and commodity.

3b. Exchange Rate Updates
Fetching Exchange Rates:
Retrieves daily exchange rates for specified currency pairs.
Uses the Close price as the most recent exchange rate for all days other than today. uses the latest available price for today.
Statuses:
Outputs the status of each exchange rate fetch:
✓ New Data: Successfully fetched live data.
# Cached: Used previously fetched data.
✗ Failed: Could not retrieve the data (e.g., network issues or invalid symbol).
Validation:
Ensures that all tickers with non-GBP currencies have corresponding exchange rates.

4. Caching:
STores historical ticker data in cache. Today's data is not cached because it could change.
Cache stored in a JSON-based cache file for reuse, named to reflect it's contents.
Checks and uses cache first before making API calls to avoid redundant requests.

5. Logging
Log File:
Logs all operations in a rotating log file, ensuring long-term usability without cluttering storage.
Captures detailed information such as:
WHat part of the script is about to start and when it ended.
API errors.
Cache usage.
Currency conversion calculations.
Processing summaries.
Timestamps are in iso format
Terminal Logging:
Provides user-friendly summaries and progress updates for key operations. Dates are presented as dd/mm/yyyy format. The terminal output is structured as follows:

Uses heading functions to structure terminal output into:

Debugging information.

6. Debugging Section of Terminal Output
For each ticker defined in the debugging section of YAML:
Prints the original currency, original price, revision calculation and and revised price after conversion.
Provides step-by-step details of adjustments, including:
Whether conversion was needed.
Exchange rate used for conversion.
Final price after conversion.

7. CSV Output for Quicken
Format:
Outputs a descending date ordered CSV file containing these columns in order.

Stock ticker.
Adjusted price in GBP.
Historical date in dd/mm/yyyy format.

Content:
Only includes successfully fetched and processed tickers.
Excludes tickers with failed or missing data.
Validation:
Ensures no duplicate or redundant data is written to the file.
8. Quicken Integration
CSV Automation:
Automates uploading the CSV file into Quicken using a pre-configured GUI automation script.
Ensures the file is in the correct format and location for Quicken to process.
Error Handling:
Detects issues such as Quicken being unavailable and provides next steps for the user.
9. Summary Output
Provides a comprehensive summary at the end of execution, including:
Total tickers processed.
Number of unknown or failed tickers.
Count of tickers converted by currency type (e.g., USD to GBP, CAD to GBP).
Success rate as a percentage.
Total execution time.
Paths to the saved log and CSV files.
Mimics your provided example format for terminal output.
Edge Cases Handled
Invalid or misspelled tickers.
Network timeouts or API rate limits.
Corrupted or missing cache files.
Non-trading days or weekends with missing stock data.
Unavailable exchange rates for specific currency pairs.
Outputs
Terminal:
Provides a user-friendly report with sections for exchange rates, stock prices, debugging, and a final summary.
CSV:
A clean, well-structured CSV file for Quicken import.
Log File:
A detailed rotating log file with all operations, errors, and debugging information.
